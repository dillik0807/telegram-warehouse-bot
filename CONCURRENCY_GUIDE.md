# Руководство по многопользовательской работе

## ✅ Реализованные меры защиты

### 1. Блокировки на уровне базы данных (SELECT FOR UPDATE)

Все критические операции с товарами используют блокировки строк:

- **add_product()** - блокирует строку товара при добавлении
- **remove_product()** - блокирует строку товара при выводе
- **add_arrival()** - блокирует строку товара при приходе
- **add_departure()** - блокирует строку товара при расходе

**Как это работает:**
```sql
SELECT quantity FROM products WHERE name = 'Товар' FOR UPDATE
```
Эта команда блокирует строку до завершения транзакции, предотвращая одновременное изменение.

### 2. Транзакции с откатом

Все операции обернуты в try-except блоки с автоматическим откатом при ошибках:

```python
try:
    # Операции с базой данных
    conn.commit()
except Exception as e:
    conn.rollback()
    return False, str(e)
```

### 3. Изоляция данных пользователей

- **context.user_data** - каждый пользователь имеет свой изолированный контекст
- **Сессии** - каждый Telegram ID имеет свою сессию в базе данных
- **Права доступа** - проверяются для каждого пользователя отдельно

## ⚠️ Важные замечания

### PostgreSQL уровень изоляции
По умолчанию PostgreSQL использует уровень изоляции "READ COMMITTED", который:
- Предотвращает грязное чтение
- Предотвращает потерянные обновления (благодаря FOR UPDATE)
- Гарантирует консистентность данных

### Таймауты блокировок
Если операция занимает слишком много времени, PostgreSQL может выдать ошибку блокировки.
В этом случае пользователь увидит сообщение об ошибке и может повторить операцию.

## 📊 Рекомендации для администраторов

1. **Мониторинг блокировок**
   ```sql
   SELECT * FROM pg_locks WHERE NOT granted;
   ```

2. **Просмотр активных транзакций**
   ```sql
   SELECT * FROM pg_stat_activity WHERE state = 'active';
   ```

3. **Настройка таймаутов** (в config.py или переменных окружения)
   ```python
   # Таймаут блокировки (по умолчанию нет)
   lock_timeout = 30000  # 30 секунд
   ```

## 🚀 Производительность

### Оптимизация для большого количества пользователей

1. **Connection pooling** - рассмотрите использование пула соединений:
   ```python
   from psycopg2 import pool
   connection_pool = pool.SimpleConnectionPool(1, 20, DATABASE_URL)
   ```

2. **Индексы** - убедитесь что есть индексы на часто используемых полях:
   ```sql
   CREATE INDEX idx_products_name ON products(name);
   CREATE INDEX idx_arrivals_date ON arrivals(arrival_date);
   CREATE INDEX idx_departures_date ON departures(departure_date);
   ```

3. **Кэширование** - для справочных данных (склады, фирмы, клиенты)

## 📝 Логирование

Все операции логируются с указанием пользователя:
- created_by - ID пользователя, создавшего запись
- created_at - время создания записи

Это позволяет отследить кто и когда выполнил операцию.

## 🔧 Устранение проблем

### Проблема: "Deadlock detected"
**Причина:** Две транзакции блокируют друг друга
**Решение:** PostgreSQL автоматически откатит одну из транзакций. Пользователь может повторить операцию.

### Проблема: "Lock timeout"
**Причина:** Операция ждет блокировку слишком долго
**Решение:** Увеличить таймаут или оптимизировать запросы

### Проблема: "Connection pool exhausted"
**Причина:** Слишком много одновременных подключений
**Решение:** Увеличить размер пула или оптимизировать закрытие соединений

## 📞 Поддержка

При возникновении проблем с многопользовательской работой:
1. Проверьте логи бота
2. Проверьте логи PostgreSQL
3. Проверьте активные блокировки в базе данных
4. Убедитесь что все соединения корректно закрываются
